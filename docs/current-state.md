# CLARIXO Current State

This document describes the current product mapping and capability status of CLARIXO under the Dual-Sided Protection Theory.

It separates:

- what is already live and usable,
- what is structurally in place,
- and what is still being closed at the edge.

---

## 1. Positioning

CLARIXO is being developed as **dual-sided infrastructure for AI producers**:

- **LLM-side protection before execution**
- **User-side protection after behavior enters the real world**

This means CLARIXO is not being defined as a single-point tool.

It is being developed as a structured layer between AI runtime behavior and real-world organizational consequence.

---

## 2. Current LLM-side mapping

The LLM-side currently maps to:

- Runtime
- AgentOps
- Approval Gateway

### Runtime
Runtime is the visible operational layer for routing, guardrails, fallback awareness, and execution-side observability.

This is the layer where CLARIXO begins to make execution paths more visible before behavior becomes real-world consequence.

### AgentOps
AgentOps is the review and operational reading layer on the LLM-side.

It currently expresses:

- row-level runtime reading,
- grouped review lanes,
- run detail explanation,
- workbench entry,
- operational review structure before commit-boundary governance.

This means runtime behavior is no longer treated as only hidden system output.  
It becomes something operators can read, group, inspect, and hand forward.

### Approval Gateway
Approval Gateway is the deepest governance layer currently defined on the LLM-side.

It is positioned as:

- the commit-boundary truth surface,
- the approval persistence layer,
- the continuation relock layer,
- the governed execution handoff layer.

Approval Gateway is not positioned as a generic admin page.  
It is positioned as the strongest pre-execution governance surface in the current CLARIXO structure.

---

## 3. Current User-side mapping

The User-side currently maps to:

- Evidence API
- Observation surfaces
- Audit Workspace

### Evidence API
Evidence API is the foundation layer for preserved AI behavior evidence.

It currently supports:

- self-serve trial project and API key bootstrap,
- evidence write path,
- evidence read path,
- project-bound evidence visibility.

CLARIXO treats evidence as a foundational requirement rather than a luxury add-on.

### Observation surfaces
CLARIXO currently exposes both LLM-side and User-side observation surfaces.

These observation surfaces are important because they move behavior visibility outward:

- runtime-side observation is no longer purely conceptual,
- behavior evidence is no longer purely internal,
- organizations do not need to wait for failure before first seeing the system.

### Audit Workspace
Audit Workspace is the structured review layer on the User-side.

It currently expresses:

- case opening,
- grouped evidence reading,
- reviewer context,
- case detail reading,
- structured export generation.

This means AI behavior is no longer only stored.  
It can now become a formal review object.

## 6. What is still being closed

CLARIXO is not being presented here as fully complete in every operational detail.

The main remaining closure area is:

### Approval Gateway final live entry closure
Approval Gateway is already structurally defined and partially validated as the deepest LLM-side governance layer.

However, its final live entry path still depends on the complete approval/session-seeded entry chain being fully closed in practice.

So the correct current description is:

- the governance structure is already clear,
- much of the surrounding chain is already validated,
- but the final Approval Gateway entry closure is still the main tail area.

---

## 7. Why this matters

The purpose of this current-state document is to prevent confusion between:

- a complete finished system,
- a live but still advancing infrastructure layer,
- and a purely conceptual framework.

CLARIXO is not only a concept.  
It already has real product surfaces.

At the same time, CLARIXO is still an advancing infrastructure layer rather than a fully finished endpoint in every direction.

That distinction matters.
